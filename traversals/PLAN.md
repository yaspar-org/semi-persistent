# Partitioned Per-Type Arenas for `rec_family!`

## Goal

Add a second macro (`partition!`) that takes a family defined by `rec_family!` and generates per-type arenas in a single container with transparent cross-arena traversals. `rec_family!` stays unchanged — the coproduct path remains available. Users choose which layout they want.

## Current State

`rec_family!` generates:

- Per-sort enums parameterized over type variables for cross-sort references
- A coproduct enum wrapping all sorts, with `sort_of()`, `dispatch()`, `multi_map`, `multi_map_ref`
- `impl Functor<R>` on the coproduct
- `ResultFam`, `__Phantom`
- All traversals in `lib.rs` operate on a single `Arena<Coproduct>`

## Design

### Two macros, user chooses

```rust
// 1. Define the family (unchanged)
rec_family! {
    pub family Lang;
    enum Expr { Var(u32), Block(Stmt), If(Expr, Stmt, Stmt), BinOp(Expr, Expr) }
    enum Stmt { Assign(u32, Expr), Seq(Variadic<Stmt>) }
}

// 2a. Coproduct arena (existing behavior, still works)
let arena: Arena<Lang<Id>> = Arena::new();

// 2b. Partitioned arenas (new, opt-in)
partition!(Lang => LangStore);
let store = LangStore::new();
```

### What `partition!` generates

**Typed ID newtypes:**
```rust
pub struct ExprId(pub usize);
pub struct StmtId(pub usize);
pub enum LangRoot { Expr(ExprId), Stmt(StmtId) }
```

**Concrete node types** (cross-sort edges resolved to typed IDs):
```rust
pub enum ExprNode { Var(u32), Block(StmtId), If(ExprId, StmtId, StmtId), BinOp(ExprId, ExprId) }
pub enum StmtNode { Assign(u32, ExprId), Seq(Variadic<StmtId>) }
```

**Mapped companion enums** (children replaced by algebra results):
```rust
pub enum ExprNodeMapped<AE, AS> { Var(u32), Block(AS), If(AE, AS, AS), BinOp(AE, AE) }
pub enum StmtNodeMapped<AE, AS> { Assign(u32, AE), Seq(Variadic<AS>) }
```

**`map_children`** per sort (one closure per child sort):
```rust
impl ExprNode {
    pub fn map_children<AE, AS>(
        &self,
        fe: impl Fn(&ExprId) -> AE,
        fs: impl Fn(&StmtId) -> AS,
    ) -> ExprNodeMapped<AE, AS> { ... }
}
```

**Container struct** with per-sort arenas:
```rust
pub struct LangStore {
    pub exprs: Arena<ExprNode>,
    pub stmts: Arena<StmtNode>,
}
```

**Multi-sort fold orchestrator** — one worklist, N-tagged tasks, N memo tables:
```rust
impl LangStore {
    pub fn fold<AE: Clone, AS: Clone>(
        &self,
        root: LangRoot,
        alg_e: impl Fn(ExprNodeMapped<AE, AS>) -> AE,
        alg_s: impl Fn(StmtNodeMapped<AE, AS>) -> AS,
    ) -> LangFoldResult<AE, AS> { ... }
}
```

**Per-sort memoization strategy selection:**
```rust
store.view()
    .with_expr_strategy::<memo::Map>()
    .with_stmt_strategy::<memo::None>()
    .fold(root, alg_e, alg_s);
```

**Per-sort `HasVariadic`** with typed pools. **Per-sort `Functor<SortId>`** so single-sort `Arena` machinery in `lib.rs` works unchanged per arena.

### Semi-persistence

Family-level snapshot = tuple of per-arena epochs. Independent rollback per sort.

## Implementation Steps

### Phase 1: `partition!` macro skeleton

1. Parse `partition!(FamilyName => StoreName)`. Extract sort definitions from the family (via a generated trait or by re-parsing the family's token output).

2. Generate typed ID newtypes, concrete node enums, mapped companion enums, `map_children`.

3. Generate the container struct with `push_*`, `get_*`, `len_*`, `mark`, `restore`.

### Phase 2: Multi-sort traversals

4. Generate `ChildBuf` (one `SmallVec` per sort) and `children_into` per sort.

5. Generate multi-sort `fold`, `fold_all`, `para`, `transform` on the container. N-tagged task enum, N memo tables, `map_children` with per-sort memo lookups on `Eval`.

6. Generate `LangStoreView<'a, ME, MS, ...>` with per-sort strategy builder methods.

### Phase 3: Extended schemes

7. Port remaining schemes: `histo`, `zygo`, `unfold`, `refold`, `zipper` to multi-sort variants.

## Testing Strategy

- `rec_family!` tests unchanged — coproduct path unaffected
- New `partition!` tests: same family, same algebras, verify identical fold results between coproduct and partitioned layouts
- Benchmark: fold throughput on coproduct vs partitioned for families with lopsided sort sizes


## Status

All 26 testorial chapters ported to the partitioned layout — see
`traversals/tests/partition_testorial.rs`. Schemes generated by `partition!`:

- `fold`, `fold_all`, `fold_ref` (via `fold_all` variants)
- `fold_short`, `fold_with_history`, `fold_with_aux`, `fold_with_original`
- `fold_pair`, `prefold`
- `unfold`, `unfold_short`, `postunfold`
- `para`, `histo`, `zygo`
- `transform`, `rewrite`, `rewrite_down`
- `Zipper`, `ZipperMut`, `ZipperCow`

Arena-free schemes (`refold`, `refold_with_history`) remain free functions —
they don't need the partition machinery.
